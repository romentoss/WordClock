Temporal API vs Date JavaScript (Stage 3 Proposal - No es estándar aún)

// Backend Deno + Temporal API - devuelve JSON: { "times": [ { "city": "Madrid", "country": "España", "timezone": "Europe/Madrid", "hour": 19, "minute": "56", "second": "22", "dayPeriod": "PM", "date": "miércoles, 24 de junio de 2026" } ] }
Característica Date (❌ Problemático) Temporal (✅ Propuesta TC39 Stage 3)
Inmutable ❌ Mutable: fecha.setDate() modifica el original ✅ Inmutable: fecha.add({days:1}) retorna nuevo objeto
Zonas horarias ❌ Solo UTC, manejo manual de offset ✅ IANA built-in: withTimeZone("Asia/Tokyo")
Cálculos ❌ Milisegundos desde epoch ✅ Precisión en cálculos, almacenamiento en ms
API ❌ Confusa: getDate(),getDay(),getUTCDate()... ✅ Clara: .day, .month, .year separados
Tipos de fecha ❌ Solo Date (mezcla date+time+timezone) ✅ PlainDate, PlainTime, PlainDateTime, ZonedDateTime, Instant
Duraciones ❌ Matemáticas con timestamps ✅ Temporal.Duration con .hours, .minutes...

Node.js vs Deno Runtime

Característica Node.js Deno
TypeScript ⚠️ Necesita build (tsc) o herramientas como ts-node ✅ Nativo: corre .ts directamente, zero-config
Dependencies package.json + npm/pnpm/yarn install ✅ Importas de URLs (esm.sh, jsr, npm:)
Permisos ⚠️ Acceso total por defecto al sistema ✅ Sandboxed: --allow-net, --allow-read, --allow-env
ES Modules ⚠️ Soporta ESM y CommonJS, puede ser confuso ✅ Solo ES Modules nativos
Tooling ⚠️ Necesitas npm install -g para CLI tools ✅ Built-in: deno test, fmt, lint, compile
Web APIs ⚠️ fetch disponible pero no todas las APIs web ✅ fetch, WebSocket, Workers, Streams estándar
npm compatibility ✅ Nativo ✅ Deno 2.x tiene first-class npm compatibility
